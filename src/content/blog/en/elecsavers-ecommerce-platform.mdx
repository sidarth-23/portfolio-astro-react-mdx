A client came to me with a familiar request: they wanted to test whether an electronics store could work for their business, but they weren't ready to commit to a full enterprise build. They needed something real enough to show customers and investors — real products, real prices, a working cart, and an admin interface where they could manage inventory and watch sales. They also wanted authentication with roles, so different team members could do different things, and a payment flow that could be swapped for a real provider the moment they decided to go live.

The catch? They wanted all of this without paying for servers that might sit idle. If no one visited, the bill should be close to zero. If a campaign drove traffic, it should scale without anyone waking up at 3 AM to restart a server.

That brief became **ElecSavers**.

## The Constraint: Credibility on a Shoestring

Most "test the waters" prototypes fail because they look like prototypes. Customers don't trust a store that feels half-built, and a client can't learn anything meaningful from a demo that no one would actually buy from. At the same time, building a full Shopify-grade platform before validation is expensive overkill.

The goal was a narrow band: **credible enough to convert, cheap enough to abandon.**

That shaped every decision. I needed a stack that could:

- Run essentially for free at low traffic.
- Scale horizontally without manual ops work.
- Let me ship a real admin panel, real inventory, real invoices, and real GST calculations for the Indian market.
- Keep the payment provider as a pluggable piece, ready to be turned on.

---

## The Stack Choice

I ended up with a serverless-first, edge-friendly stack:

- **Framework**: [Next.js 15](https://nextjs.org/) App Router for server-side rendering, API routes, and server actions in one deployable unit.
- **Database**: [Turso](https://turso.tech/) (LibSQL) — a managed SQLite database replicated to the edge. SQLite is famously small and fast; Turso makes it serverless.
- **ORM**: [Drizzle ORM](https://orm.drizzle.team/) for type-safe queries and minimal cold-start overhead.
- **Auth**: [Better Auth](https://better-auth.com/) for email/password, Google OAuth, and role-based access control.
- **UI**: Tailwind CSS v4 + [shadcn/ui](https://ui.shadcn.com/) for a clean, consistent interface without reinventing components.
- **Tooling**: [Bun](https://bun.sh/) for fast installs and script execution.

The reasoning was simple. Next.js on Vercel starts at $0. Turso's starter tier is generous enough for this stage. SQLite keeps the data model simple. Better Auth avoids a separate auth service. The combined monthly cost can be effectively zero while the site is quiet, and the architecture can absorb spikes without re-platforming.

## The Data Model: Just Enough Schema

E-commerce schemas can balloon quickly. I deliberately kept the core small: products, orders, and the relationships needed for multi-vendor support and Indian tax compliance. Drizzle made it easy to express this in type-safe SQLite.

```typescript filename="schema.ts"
import { sqliteTable, text, integer, real } from "drizzle-orm/sqlite-core"

export const products = sqliteTable("products", {
  id: integer("id").primaryKey(),
  name: text("name").notNull(),
  price: real("price").notNull(),
  category: text("category").notNull(),
})

export const orders = sqliteTable("orders", {
  id: integer("id").primaryKey(),
  userId: text("user_id").notNull(),
  total: real("total").notNull(),
  status: text("status").notNull().default("pending"),
})
```

The query layer stayed equally minimal. Because Drizzle knows the schema types at compile time, the frontend gets the same guarantees the database does.

```typescript filename="queries.ts"
import { eq } from "drizzle-orm"
import { db } from "./db"
import { products, orders } from "./schema"

export async function getProductById(id: number) {
  return db.select().from(products).where(eq(products.id, id)).get()
}

export async function getOrdersByUser(userId: string) {
  return db.select().from(orders).where(eq(orders.userId, userId)).all()
}
```

This isn't the whole schema — vendors, stock levels, GST rates, and invoices all have their own tables — but the core idea is the same: keep the model lean enough that a single SQLite file on Turso can handle it comfortably, and rich enough that the business logic is real.

## The Admin Panel: Where the Business Actually Runs

The storefront is what customers see, but the admin panel is where the client decides whether the experiment is working. I can't show screenshots of the live admin because it's behind authentication, but the structure is straightforward.

On signing in, an admin lands on a dashboard with counts for products, orders, customers, and revenue. To keep these numbers snappy, I avoided running expensive `COUNT(*)` queries on every page load. Instead, the dashboard uses a cached-count table that updates when products or orders change. It costs a little more logic upfront and saves a lot of perceived slowness later.

The sidebar gives access to:

- **Products**: add, edit, or deactivate inventory, set prices and featured flags, manage vendor stock references.
- **Categories**: organize the catalog into the groups customers filter by.
- **Vendors**: track the South Indian supplier partners whose stock the store surfaces.
- **Orders**: see what customers have bought, update statuses, and download invoices.
- **Customers**: view registered users and their order history.

Every action is gated by roles. Better Auth handles the session, and the role field on each user decides whether they can access `/admin` at all, and what they can touch once inside. A support user might read orders; an operations user can edit products; only an admin can manage vendors and commissions.

## The Storefront: Real Enough to Trust

The public site is built to look like a store that has been around for a while. The homepage leads with value props — free shipping over ₹500, secure payment, easy returns — and a featured-products section. The products page supports search, category filters, brand filters, price ranges, and grid or list views. Product pages include specs, HSN codes, GST rates, and related products.

The cart persists across sessions for logged-in users. At checkout, the system calculates CGST, SGST, or IGST depending on the shipping state and generates a PDF invoice using `@react-pdf/renderer`. The numbers are real, the tax logic is real, and the invoice format is ready for actual fulfillment.

The payment step is intentionally left as a wired-but-swappable integration. The client could plug in Razorpay, Stripe, or another provider without rewriting the order flow. Until then, the prototype demonstrates the full purchase path end to end.

## Developer Experience: Seed, Build, Iterate

A prototype only helps if you can reason about it quickly. I included a `db:seed:real` script that populates the database with 68 authentic electronics products, realistic vendor references, sample orders, and categories. The first time the client opens the site, it looks lived-in rather than empty.

End-to-end type safety from the schema through the server actions to the UI components means refactors are cheap. When the client inevitably asks for a new field or a new report, the compiler catches most mistakes before they reach the browser.

---

## What This Actually Is

ElecSavers is not a finished, enterprise-grade marketplace. It is a **validated starting point**: a real store front, a real admin back end, real inventory and invoicing, and a payment slot waiting for the client's go-live signal.

The bet is that the client can run real traffic through it, learn whether the electronics niche works for their audience, and make an informed decision about whether to invest in the next phase — without having paid for infrastructure they didn't need while they were still guessing.

That, to me, is the right shape for a prototype: credible enough to teach you something, affordable enough to throw away if the lesson is "not yet."
