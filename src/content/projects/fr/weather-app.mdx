Every developer has built a weather app at some point. It’s the “Hello, World” of API consumption — a contained scope, a free public API, and just enough UI complexity to be interesting. I built mine to get more comfortable with Next.js’s App Router and to see how quickly I could go from zero to a deployed, interactive UI.  
   
Here’s how it came together.

---

## The Stack

- **Framework:** Next.js (App Router)
- **Language:** TypeScript
- **Styling:** Tailwind CSS + shadcn/ui
- **API:** WeatherBit
- **Deployment:** Vercel

Nothing exotic. The point was to use tools I was already working with and focus on the problem — fetching, shaping, and presenting weather data cleanly.

---

## Why WeatherBit?

There are plenty of weather APIs out there — OpenWeatherMap is the most common, Tomorrow.io is more feature-rich, Open-Meteo is fully free with no key required. I went with WeatherBit because it has a clean REST API, a generous free tier, and the response shape is straightforward to work with in TypeScript. One API key in `.env`, one fetch call, structured data back.

```plaintext
WEATHER_BIT_API_KEY=your_key_here
```

That’s the entire environment setup. If you’re cloning the repo, that’s all you need.

---

## How It Works

The app takes a city name as input, hits the WeatherBit current weather endpoint, and renders the response as a UI. The key data points surfaced:

- Current temperature
- Feels-like temperature
- Weather condition and description
- Wind speed
- Humidity
- UV index
- City and country display  
The interaction is intentionally simple: type a city, get the weather. No fancy autocomplete, no maps, no multi-day forecast — just the current conditions rendered cleanly.

---

## Next.js App Router and API Routes

One thing worth calling out: the WeatherBit API key never hits the client. The fetch happens through a Next.js API route, which acts as a thin proxy between the browser and the WeatherBit endpoint. The client calls `/api/weather?city=Chennai`, the route handler picks up the request, makes the authenticated call to WeatherBit server-side, and returns the shaped data.

```typescript
// src/app/api/weather/route.ts
export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const city = searchParams.get("city");
 
  const res = await fetch(
    `https://api.weatherbit.io/v2.0/current?city=${city}&key=${process.env.WEATHER_BIT_API_KEY}`
  );
 
  const data = await res.json();
  return Response.json(data);
}
```

This is the pattern you’d use in production too — keep secrets on the server, expose only what the client needs. The App Router makes this trivially clean: one file, one exported function, done.

## The UI with shadcn

Shadcn’s component library keeps the UI consistent without a lot of custom CSS. The weather card is built around shadcn’s `Card` component, with Tailwind handling spacing and layout. The input and search trigger use shadcn’s `Input` and `Button` components.  
   
The result is a UI that looks polished without requiring a design system to be built from scratch — which is exactly what you want for a project like this where the interesting work is in the data flow, not the visual layer.

---

## What I Took Away From This

A few things became clearer from building this:  
   
**API routes as a privacy layer.** It’s easy to accidentally leak an API key in a client-side fetch. The App Router’s route handlers make the server-side pattern so low-friction that there’s no excuse to do it the wrong way.  
   
**TypeScript pays off fast with API responses.** Typing the WeatherBit response shape — even partially — made the component code significantly cleaner. Once you have a typed `WeatherData` interface, the rest writes itself.  
   
**shadcn is worth having in your default setup.** Even for a project this small, having accessible, well-styled components as a starting point beats writing card and input styles from scratch every time.
