I’ve been wanting to properly try out [Astro](https://astro.build) for a while. Not for a portfolio, not for a blog — those are the obvious ones. I wanted something that felt like a real client site: multiple pages, components, images, a contact section. A dental clinic website was the perfect excuse. It’s the kind of project a local business would actually commission, and it gave me a concrete scope to work within.  
   
The result? Genuinely surprised by how smooth and fast the output was. Here’s the breakdown.

---

## Why Astro for This?

Astro’s whole pitch is **zero JS by default**. It ships pure HTML and CSS to the browser unless you explicitly opt into interactivity. For a static marketing site like a clinic homepage, that’s exactly what you want. No React hydration overhead, no bundle to parse — just pixels on screen, fast.  
   
I also wanted to test Astro’s **View Transitions API** support. Astro has first-class support for the browser-native View Transitions API, which gives you smooth, animated page navigations that feel app-like — without a single line of JavaScript router code.  
   
DaisyUI was the natural pairing for component styling. It sits on top of Tailwind, gives you semantic class names (`btn`, `card`, `hero`, `badge`), and has a built-in theming system that made it trivial to set up a clean, professional colour palette for the clinic.

---

## The Stack

- **Framework:** Astro
- **CSS:** Tailwind CSS + DaisyUI
- **Content:** Static `.astro` pages with hardcoded/mock data
- **Deployment:** Vercel
- **Language:** TypeScript No CMS, no backend. Pure static output. The goal was always performance and smoothness, not a feature-complete admin system.

---

## What’s on the Site

A typical dental clinic website needs a handful of core sections. Here’s what I built out:  
   
**Home / Hero** — A full-width hero section with the clinic name, tagline, and a primary CTA to book an appointment. DaisyUI’s `hero` component made this trivially easy to structure.  
   
**Services** — A grid of cards covering common treatments: checkups, fillings, root canals, teeth whitening, braces, and implants. Each card uses the DaisyUI `card` component with an icon and a short description.  
   
**About** — A two-column layout with clinic info and a doctor profile. This is where the DaisyUI `avatar` and `badge` components shine — clean, semantic, no custom CSS needed.  
   
**Testimonials** — A carousel/grid of patient reviews. DaisyUI’s `carousel` component gave me a nice horizontal scroll out of the box.  
   
**Contact / Appointment** — A simple form section with the clinic’s address, phone, and hours. DaisyUI form components (`input`, `textarea`, `btn`) keep the styling consistent throughout.

---

## Why DaisyUI Worked Really Well Here

For a content site like this, DaisyUI is almost unfairly productive. A few things that stood out:  
   
**Theming is first-class.** You pick a named theme (or define your own), and every component inherits it automatically. I used the `corporate` theme as a base and tweaked the primary colour to a soft teal that felt appropriate for a healthcare context. Zero per-component colour overrides needed.  
   
**Semantic class names read like English.** Looking at `<div class="card card-compact shadow-lg">` in your markup actually tells you what it is. Compared to raw Tailwind utility soup, the components stay readable.  
   
**It doesn’t fight you.** When I needed to break out of the component and add a custom layout, Tailwind utilities were right there. DaisyUI sits on top cleanly — it doesn’t lock you in.

---

## The Performance Story

This is the part I was most curious about. Astro’s output for a site like this is essentially **zero-JavaScript HTML**. Here’s what that looks like in practice:

- **Lighthouse score:** 100 Performance, 100 Best Practices on every page
- **LCP (Largest Contentful Paint):** Under 1s on desktop
- **Total JS bundle sent to browser:** ~0 KB (for pages without interactive islands)
- **CLS:** 0 — layout is fully resolved before paint since there’s no hydration step The View Transitions made page navigation feel buttery. Moving from the Home page to Services to Contact had a smooth crossfade that made it feel like an SPA, but with none of the SPA overhead. That contrast — SPA feel, static site performance — is Astro’s killer feature in practice.

```plaintext

├── components/

│   ├── ServiceCard.astro
│   ├── Testimonial.astro

├── layouts/
│   └── BaseLayout.astro       # View Transitions enabled here
├── pages/
│   ├── index.astro            # Home
│   ├── services.astro
│   ├── about.astro
│   └── contact.astro
└── styles/
    └── global.css             # Tailwind + DaisyUI imports
```

Astro’s file-based routing meant each page was just a `.astro` file. No router config, no wrapping providers. The `BaseLayout` component wraps every page and is where I enabled View Transitions with a single directive.

## What I’d Do Differently

A few things I’d revisit if this went beyond a sideproject:

- **CMS integration** — Astro has excellent support for content collections and headless CMSes like Sanity or Contentful. For a real clinic, the doctor should be able to update services and hours without touching code.
- **Appointment booking** — The contact form is static right now. Integrating something like Cal.com or a simple Supabase form submission would make it actually functional.
- **i18n** — Astro has built-in i18n routing. Given that this is a local clinic context (Tamil Nadu), adding a Tamil language version would be a genuinely useful exercise.
- **Image optimisation** — Astro’s `<Image />` component handles this automatically. I used it for some assets but not consistently; going all-in would push LCP even lower.

---

## Final Thoughts

This was exactly what I wanted from a sideproject — a constrained, real-world scope that let me focus on the _technology experience_ rather than product decisions. Astro delivered. The DX was smooth, the output was fast, and the View Transitions feature alone made the site feel polished in a way that would have required a full SPA framework before.  
   
If you’ve been looking for an excuse to try Astro, a local business landing page is a perfect fit. Scope it, style it with DaisyUI, deploy it to Vercel in minutes, and then stare at a Lighthouse score you’ve never seen before.
